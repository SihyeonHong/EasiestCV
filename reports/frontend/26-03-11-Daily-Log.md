## 3. 아키텍처 결정 사항 (Architecture Decisions)

### 3-1. `useCheckEmail` 훅의 설계 방식 (동적 콜백 vs 내부 응집)

`useCheckEmail` 훅은 컴포넌트 내부에서 라우팅이나 전역 상태를 직접 제어하는 다른 훅(예: `useLogin`, `useSignUp`)과는 달리, 파라미터로 콜백(`UseCheckEmailProps`)을 주입받는 형태로 설계되었습니다. 향후 코드 복기 시 혼동을 방지하기 위해 다음과 같이 설계 사유를 기록합니다.

- 컴포넌트 내부 응집 (Dumb Component & Smart Hook)
  - UI 컴포넌트를 완전히 시각적 요소로만 남기고 비즈니스 로직을 모두 훅으로 분리하는 방식입니다.
  - "UI 컴포넌트는 UI만 그려야 한다"는 원칙은 대규모 재사용이 필요한 버튼, 인풋 같은 순수 시각적 요소에나 적합합니다. SignUpCard처럼 재사용성이 낮고, 페이지의 큰 부분을 차지하며 특정 프로세스를 대변하는 핵심 도메인 컴포넌트는 폼 상태와 비즈니스 흐름 제어를 훅과 함께 응집하는 것이 자연스럽습니다.
- 동적 콜백 주입 방식 (현재 채택된 방식)
  - useCheckEmail은 단순 데이터 페칭과 에러 핸들링이라는 단일 책임만 수행합니다.
  - 데이터 조회 결과를 컴포넌트(SignUpCard)로 돌려주어, 상위 도메인 컴포넌트가 모달을 띄울지 다음 단계로 넘어갈지 등의 국소적 UI 제어권을 직접 쥐도록 위임(Inversion of Control)했습니다. 이를 통해 훅의 기능을 무겁게 만들지 않으면서도 독립성을 확보했습니다.
