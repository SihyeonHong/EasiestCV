# 이메일 중복 확인 UX 검토

## 1. "중복 확인" 영문 번역 제안

### 배경 및 번역 후보 비교

현재 "중복 확인" 버튼의 영문 레이블은 "Check"로 임시 적용되어 있습니다. 하지만 "Check"라는 단어는 "중복 확인"이라는 한글 표기와 비교했을 때, 사용자가 기능을 예측하기 어려운 모호한 표현입니다.

이를 구체화하기 위해 일반적인 서비스에서는 계정이나 이메일 중복 확인 시 "Check availability(사용 가능 여부 확인)"라는 표준 용어를 주로 채택합니다. 그러나 **우리 서비스에서는 이 표현을 사용할 수 없습니다.** `userid`를 고유 키로 채택하여 동일한 이메일로 여러 계정을 생성하는 것이 "이미 허용된(Available)" 상태이기 때문입니다.

이 기능의 실제 목적은 회원가입을 막기 위해서가 아니라, **"가입 사실을 잊고 새로 가입하려는 사용자에게 기존 계정의 존재를 알려주고 어떻게 할지 묻기 위함"**입니다. 이 맥락을 바탕으로 여러 번역 후보를 비교했습니다.

| 후보               | 장점                                                                                            | 단점                                                                      |
| ------------------ | ----------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------- |
| Check              | 텍스트 길이가 짧음                                                                              | 구체적으로 무엇을 확인하는지 목적어가 불충분하여 인지 부하 발생           |
| Check availability | 웹 표준 가입 폼에서 사용되는 가장 보편적이고 익숙한 형태                                        | 이메일 중복 가입을 허용하는 본 서비스의 실제 정책 및 로직과 의미가 충돌함 |
| Look up            | "기록이나 데이터베이스에서 정보를 찾아보다"라는 뜻으로 기존 가입 이력 조회라는 실제 역할에 맞음 | 특별한 단점 없음. "Availability"의 이분법적 판단을 배제함.                |

### 제안

**"Look up"**을 권장합니다.
"Availability"처럼 사용 가능 여부를 판별한다는 오해를 주지 않으면서도, "이메일을 기반으로 기존 가입 정보(계정)를 찾아본다"는 서비스의 실제 의도를 가장 명확하고 자연스럽게 전달하는 표현입니다. 버튼을 클릭하여 확인이 끝나면 "Checked" 대신 "Done" 혹은 "Looked up"으로 전환하면 잘 어울립니다.

## 2. 기존 계정 안내 UI 방식 비교

### 전제 조건

사용자가 이미 등록된 이메일을 입력했을 경우, 이를 '에러'로 취급해 입력을 막는 것이 아닙니다. 대신 "당신의 이메일로 가입된 기존 아이디(userid)가 있는데, 기존 계정으로 로그인할 것인지 아니면 원래대로 새 계정을 만들 것인지" 정보와 선택지를 안내해야 합니다.

이 목적을 달성하기 위한 두 가지 UI/UX 패턴을 비교했습니다.

### 방식 1: 명시적 버튼 클릭 + AlertDialog (원래 계획)

이메일 필드 옆의 버튼("Look up")을 명시적으로 클릭했을 때, 중복 이메일이 발견되면 화면에 AlertDialog를 띄워 다음 행동(로그인 이동, 비밀번호 찾기, 계속해서 새 계정 만들기)을 안내하는 방식입니다.

- **장점**:
  - **사용자의 기대와 피드백의 일치**: 사용자가 "확인해볼게"라는 의도로 버튼을 클릭했으므로, 다이얼로그가 나타나도 "자신이 요청한 결과"로 인식하여 덜 당황합니다.
  - **강력한 포커스**: 폼 작성을 계속할지 기존 계정으로 돌아갈지를 묻는 중요한 의사 결정 구간에서, 화면을 가리는 다이얼로그는 사용자의 이목을 집중시켜 확실한 선택을 유도합니다.
- **단점**:
  - 확인 버튼을 굳이 찾아 눌러야 하므로 폼 입력 흐름에서 클릭 비용이 1회 늘어납니다.

### 방식 2: onBlur + Inline UI

사용자가 이메일을 입력하고 탭이나 마우스 클릭으로 다음 칸으로 넘어갈 때(onBlur) 백그라운드에서 조회 API를 호출하고, 중복이 있다면 이메일 입력 칸 아래에 인라인(Inline) 형태로 안내 문구와 링크들을 노출하는 방식입니다.

- **장점**:
  - 확인 버튼 자체가 사라져 인풋 컴포넌트가 깔끔해지고, 사용자는 폼을 끊김 없이 쭉 채워 나가는 듯한 느낌을 받을 수 있습니다.
- **단점**:
  - **복잡한 정보의 레이아웃 한계**: 일반적으로 인라인 UI는 "비밀번호 형식이 틀렸습니다" 같은 한 줄짜리 단순 에러 메시지를 주기에 적합합니다. 반면, 이 기능은 기존 아이디 노출, 로그인 링크, 비밀번호 초기화 링크 등 복합적인 정보를 담아야 하므로 좁은 인라인 영역(Input들 사이)에 넣기에는 시각적 위계가 크게 깨지고 폼의 상하 높이가 심하게 요동치게 됩니다.
  - **시선 분산과 당황**: 사용자의 시선은 이미 다음 칸인 "비밀번호 입력"으로 향해 있는데, 이전 칸 아래에서 복잡한 내용이 갑자기 튀어나와 폼이 늘어나면 맥락이 끊기고 혼란을 가중시킵니다. 인라인에서 기존 계정 정보를 줘도 이를 무시하고 비밀번호 칸을 계속 채우는 경우 UI가 다소 어색해집니다.

### 결론 및 권장 사항

**"명시적 버튼 클릭 + AlertDialog(원래 계획)" 방식을 권장합니다.**

이메일 중복 안내 처리는 단순한 형식 검증이 아니라, **진행 중인 회원가입 플로우 자체를 중단할지 판단하게 만드는 비중 높은 갈림길(Branching)**입니다.

인라인(Inline) UI 방식은 이런 비중 있는 선택지를 폼 사이에 구겨 넣어야 하므로 시각적인 정돈(Aesthetics) 관점에서 매우 불리합니다. 따라서 사용자가 먼저 의도를 밝히고(버튼 클릭) 그에 맞추어 확실한 포커스 영역(AlertDialog)에서 옵션을 제시하는 편이 훨씬 직관적이고 안정감 있는 UX를 제공합니다. 다이얼로그에서 "새 계정 만들기" 버튼을 누르면 모달이 부드럽게 닫히면서 이메일이 "확인 완료" 상태로 남는 흐름이 가장 이상적입니다.
