# 2026-03-10 Frontend Test Plan

## 1. 프론트엔드 작업 테스트 요구사항

"reports/frontend/26-03-10-Daily-Log.md"에 명시된 주요 작업을 테스트하기 위해 다음 항목들이 필요합니다.

- API 연동 테스트: "DuplicateEmailAlertDialog.tsx"에서 백엔드 API를 올바르게 호출하여 중복 여부를 파악하는지 확인합니다.
- 상태 전이 검증: 이메일 중복 확인 이후 계정 선택 흐름 및 회원가입 성공 시 노출되는 환영 메시지의 UI 상태 전이를 검증합니다.
- 반응형 레이아웃 점검: 다양한 기기 해상도 환경에 대응하는 레이아웃 및 상태 관리 로직의 결함을 점검합니다.

## 2. 수동 승인 최소화 테스트 전략

보안 상의 위험으로 터미널 자동 실행을 허용하지 않으면서도, 수동 승인의 피로도를 줄이기 위한 테스트 전략을 제안합니다.

- 단일 테스트 스크립트 통합: 에이전트가 디버깅을 위해 여러 스크립트를 파편적으로 실행하는 대신, 필요한 모든 검증이 포함된 "Unit Test" 또는 "E2E Test" 코드를 완성된 파일로 작성하여 제공합니다.
- 사용자 1회 실행: 파일 작성 완료 후, 사용자가 개발 환경 터미널에서 직접 실행 명령어를 1회 입력하여 전체 테스트를 수행하도록 안내합니다.
- 코드 정적 리뷰 강화: 스크립트를 실행하기 전에 오직 에이전트의 코드 스캐닝 능력에 기반한 정적 분석만으로 1차 검증을 완료합니다.

> **참고 (테스트 단계와의 관계)**
> 위에서 제안한 "단일 스크립트 통합 및 수동 승인 1회 실행" 방식은 특정 테스트 계층(Phase)에 속한다기보다는 **모든 프론트엔드 파편화 스크립트를 하나로 묶는 에이전트 운영/협업 전략**입니다.
> 현재 질문해주신 프론트엔드 상황에 이 방식을 대입한다면, 주로 **"Phase 3 (Headless UI 통합 테스트)"**에 적용됩니다. 즉, 모달 렌더링, 클릭, API 통신 등의 모든 흐름을 파편화된 브라우저 디버깅 스크립트 대신 `React Testing Library` 기반의 단일 통합 테스트(.test.ts) 파일로 작성하여 한 번의 터미널 명령어만으로 검증하게 됩니다.

## 3. 기존 테스트 인프라 전략 연계 및 향후 기대효과

2월 12일자 "테스트 인프라 도입 전략 보고서"(`26-02-12-Test-Infrastructure-Strategy.md`)에서 우리는 Vitest 기반의 Phase 1을 적용하였습니다. 현재 진행 중인 프론트엔드 작업에 나머지 Phase 도구들을 도입할 경우, 다음과 같은 테스트 자동화와 효율성 증대를 기대할 수 있습니다.

- Phase 2 (API Mocking 추가): 데이터베이스 없이도 이메일 중복 확인 API 엔드포인트를 고립시켜 안정적으로 단위 테스트를 수행할 수 있습니다.
- Phase 3 (Headless UI 통합 테스트 - React Testing Library): 백엔드 API가 이미 개발 완료된 현재 상황에서는 가벼운 "jsdom" 환경 위에서 "@testing-library/react"를 사용하여 UI 컴포넌트를 렌더링하고 테스트하는 것이 효율적입니다. 화면을 직접 확인하지 않고도(Headless) 요소를 클릭하고 컴포넌트의 상태 전이(`DuplicateEmailAlertDialog.tsx` 렌더링 등)를 자동으로 검증할 수 있습니다. 실제 백엔드 API 응답은 내부에서 `fetch` 모킹으로 제어하여 E2E 테스트 수준의 신뢰도를 빠르게 확보할 수 있습니다.
- Phase 4 (E2E 테스트 - Playwright/Cypress): 브라우저를 직접 띄워 시각적으로 테스트하는 이 방식은 "모바일 기기 뷰포트에서의 레이아웃 붕괴 확인" 등 브라우저 렌더링 단의 검증이 필요한 핵심 통합 시나리오에만 최소한으로 도입하여 실행 속도와 테스트 피로도를 관리합니다.
